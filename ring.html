1 Discover

Auditing Ring's web component ecosystem

Ring was migrating their website's components to a new headless CMS. The first step was auditing every existing component — cataloging inconsistencies, identifying legacy patterns that needed updating, and understanding which components were truly "global" (used across multiple pages and contexts) versus page-specific. This audit became the foundation for the new component library.

Problem Statement

The legacy CMS had accumulated years of one-off components with inconsistent styling. Developers couldn't tell which styles were current vs deprecated, leading to visual inconsistencies across Ring's web properties.

2 Ideate

Defining the global component architecture

I worked with the team to define which components should be global — navigation, hero sections, product cards, CTAs, footers — and established a hierarchy of atomic, molecular, and organism-level components. Each component needed to be flexible enough to work across Ring's diverse page types while remaining visually consistent.

📌
Replace: Component architecture / planning diagram
3 Design

Design system with detailed developer specs

I built global components using a design system in Sketch with detailed spec sheets for every property — sizing, typography, spacing, and styling. A key innovation was creating a dual-track documentation system: each component had both "legacy styles" (what currently existed) and "new styles" (the target state), allowing developers to easily identify what needed to change and follow the migration path.

🎨
Replace: Design system specs with sizing/typography/spacing (Recommended: 1600x900px)
Headless CMS Approach

By designing components that were content-agnostic and purely structural, we enabled the headless CMS to serve content dynamically while the design system ensured every rendering was visually consistent — regardless of what content was plugged in.

4 Iterate

Testing component flexibility across page types

We validated each global component against multiple page contexts — product pages, campaign landing pages, support pages — to ensure flexibility. Working directly with developers, I iterated on spacing and responsive breakpoints to handle edge cases that only surfaced when real content was plugged into the system.

🔄
Replace: Component testing across page types
5 Handoff

Legacy-to-new migration specs for developers

The handoff was uniquely structured around the migration itself. Each component's documentation included a clear "before and after" — legacy style tokens mapped to new tokens, with redlines showing exactly what changed. This allowed developers to migrate components incrementally without breaking the live site, making the entire CMS migration smooth and predictable.
